# VR controller input

Driving the head/body from the game's own look and move input, and — eventually — from VR controllers so it isn't gamepad-only in the headset. The read side (tapping and consuming the engine's input so our schemes own the look and move deltas) ships today; the write side (injecting VR controller state as native input) is not yet wired. For the underlying engine mechanism — the action effectors, the action-ID table, the write API, and the mouse/UI pipeline — see `docs/engine/input.md`.

## Tapping the look input

For VR (and the flatscreen prototype) we drive the head/body ourselves, so the game's own look input must stop rotating the camera and feed our scheme instead. We tap the engine's device poll rather than the camera pipeline. Only look and move are consumed today, out of the full action set described in `docs/engine/input.md`.

The shipped detour is on `InputDeviceManager::Update(dt)` — the per-frame device poll (`payload/src/hooks/input/look.rs`). After the original poll runs, the detour reads the four look effectors off `LocalPlayerActionMap`, clears them (zeros `m_Value` / `m_PrevValue`, sets `m_State` to `Idle`) so the game's camera system sees no look input, and feeds the net deltas into the headpose simulation via `headpose::sim::on_input_tick`. Running post-poll means the values are the fully-resolved per-frame deltas; clearing them at the effector is safe because the look effectors feed camera code only, so nulling them doesn't disturb aim assist (which derives from the resulting camera orientation). When egui captures input the detour still ticks the sim with a zero delta, so the latch and mode-detection cadence keep running.

The tick runs on the engine's fixed-rate sim tick, inside the `Update` hook, so the published pose pair rotates in phase with the engine's sub-frame interpolation reset — see `payload/src/headpose/sim.rs`.

**Why not the camera pipeline.** An earlier approach considered tapping the look path where the camera consumes it: `GameCameraManager::UpdateBlackboardValues` reads gamepad look, and `InputToOrbitModifier::CalculateInputDeltaAngles` combines the devices, applies per-axis sensitivity, and returns the final delta-angle, which its caller `InputToOrbitModifier::ProcessCameraContext` applies via `BoomTransform::DeltaTransform`. `CalculateInputDeltaAngles` is off-limits as a hook point — it carries an anti-tamper codemarker (`docs/engine/input.md`) — and while `ProcessCameraContext` itself is clean, consuming the effectors post-poll is simpler and keeps us out of the codemarker's blast radius entirely, so that is what shipped.

## Tapping the move input

Move is handled in the locomotion tasks rather than by clearing effectors (`payload/src/hooks/input/locomotion.rs`). The always-on FPS-movement shim (`MovementConfig::force_fps_movement`) forces the local player's aim flags (`Character::m_AimFlags`, `m_AimingWeapon`) only for the duration of each `NStateTask_InputLocoMoveTask` / `NStateTask_InputLocoAimRelativeTask` `Update`, and restores them immediately after. The locomotion tasks pick between run/steer and aim-relative strafe by reading those flags — the same flags that drive the weapon raise, the reticle, and auto-aim — so forcing them only across the task's `Update` makes the movement branch see "aiming" (directional strafe) while the aim system never does. Forcing the flags globally would drag the weapon raise and reticle in with it; the scoped force avoids that. The character updates on its own worker thread, so nothing else reads the flags mid-task.

`NStateTask_InputLocoSetTargetDirTask::SetupTargetDir` reads the move axes, rotates them camera-relative, and writes the move vectors to the character blackboard; the headpose body-yaw scheme and the slide overrides build on top of it (heading via the target-face-dir blackboard, displacement via `EvaluateCharacterDisplacement`). See `head-and-body.md` for the head/body coupling.

## The write path (future)

Not yet implemented. The intended design injects VR controller state as effector values so the engine reads VR input as native input — the sticks and triggers as effector axes, the buttons as effector states — gated so real-gamepad input and VR input don't fight. Locomotion direction and turning would tie into the head/body scheme (`head-and-body.md`). It drives through the write API and timing rules documented in `docs/engine/input.md` (`LocalPlayerActionMap::ForceSetPressed`/`ForceSetClicked`, injected after `InputDeviceManager::Update`).

## The mouse and the UI (issue #9)

The engine's mouse pipeline — `WndProc` → `CUIManager::SetMousePos` → `SendMouseEvents`, the client-to-movie-viewport conversion, the doubled `HCURSOR`/`MCI_cursor` sprite, `RestoreAfterReset`, and `CCommMapUI::OnManageInput`'s own conversion — is documented in `docs/engine/input.md`. This section covers what the HUD redirect breaks in it and how the mod fixes it back up.

The HUD redirect breaks the conversion: it points `m_CachedViewport*` and `m_MovieScale*` at our offscreen texture, so window-client pixels no longer relate to either space. The shipped fix (`payload/src/hud/cursor.rs`, the `SendMouseEvents` detour in `payload/src/hooks/ui.rs`) replaces the mapping while the redirect is active: normalize the client position against the window's live client rect (published from the `WndProc` detour — the back buffer can be a different size and aspect under Wine/Proton scaling), map that fraction of the stage through the movie's live `ViewportMatrix` (`HandleEvent`/`NotifyMouseState` invert exactly that matrix, so the forward map is correct whatever state the matrix is in), and hand Scaleform the whole state per frame via `MovieImpl::NotifyMouseState` (slot 37: position plus button bitmask, diffed internally into hover/press/release — which also sidesteps the DirectInput-delta gate). Buttons and the wheel are tracked from window messages in the `WndProc` detour; the `MCI_cursor` clip is parked offscreen and replaced by our own circle-dot quad floating just off the panel, visible exactly when the game's own cursor policy (`m_MouseCursorShowRefCount`) shows a cursor. Both axes normalize independently, so a window aspect different from the panel aspect stretches mouse travel rather than leaving parts of the panel unreachable.

Two auxiliary seams keep the rest of the pipeline honest:

- **The engine's reset path**: the redirect maintenance compares the movie's live `GFx::Viewport` against the redirect each render tick and re-applies on drift (`hud::binding::movie_viewport_matches`), on top of the engine's own `RestoreAfterReset` resize (`docs/engine/input.md`).
- **The map**: the redirect bypasses `m_MouseDeltaX/Y`/`m_MouseScaleFac` (X survives because its delta is zero; Y is offset by the stale half-letterbox). Generic Scaleform widgets hit-test through the injected path and never notice, so the map was the one screen left broken. A detour replaces the conversion with the same client-rect-to-stage normalization the injection uses, which fixes icon picking, click selection, drag panning, and zoom-to-cursor together.
